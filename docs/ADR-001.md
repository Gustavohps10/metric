# ADR-001 — Metric Product Architecture

## Status

Accepted

## Date

2026-03-07

---

# 1. Context

O projeto **Metric** é uma ferramenta para rastreamento e visualização de produtividade e atividade de desenvolvimento.

O objetivo do produto é oferecer:

- Cliente **Desktop** (Windows / Linux / macOS)
- Possibilidade de **Self-Hosted**
- Modelo **Open-Core**
- Planos **Pro** e **Enterprise**
- Sistema de **organizações e membros**
- **Painel web administrativo**
- Fluxo simples de autenticação para equipes grandes

O sistema deve ser capaz de atender:

- Desenvolvedores individuais
- Times pequenos (3–10 pessoas)
- Empresas grandes (30+ desenvolvedores)

Além disso, o projeto busca **minimizar complexidade operacional**, especialmente em:

- autenticação
- segurança
- gerenciamento de infraestrutura

---

# 2. Product Model

## Estratégia: Open-Core

O produto seguirá o modelo **Open-Core**.

### Definição

Open-Core significa que:

- o **núcleo do software é open source**
- funcionalidades avançadas são **proprietárias**

### Estrutura

Open Source:

- Desktop client base
- Interface principal
- Visualização de dados local
- Integração básica

Pro / Enterprise:

- Organização e times
- Painel administrativo
- Analytics avançado
- Controle de membros
- Integrações avançadas
- Gestão centralizada

### Motivo da decisão

Benefícios do modelo Open-Core:

1. Crescimento orgânico via comunidade
2. Confiança maior no produto
3. Distribuição viral
4. Monetização sustentável

Esse modelo é amplamente utilizado por empresas como:

- GitLab
- Sentry
- PostHog
- Grafana

# 3. Deployment Model

O produto terá **três modos de uso**.

## 3.1 Desktop Client

Aplicativo nativo disponível para:

- Windows
- Linux
- macOS

Fluxo:

```
Usuário baixa cliente
↓
Abre aplicação
↓
Login via Magic Link (utilizado apenas para planos pagos).

OBS: Nos planos Pro e Enterprise, o Magic Link é utilizado para autenticação e acesso às funcionalidades de organização, como controle de membros e painel administrativo.
    No plano gratuito não há gerenciamento de membros ou organizações. O usuário pode utilizar o aplicativo livremente, conectando-se às datasources disponíveis sem necessidade de autenticação.

↓
Cliente sincroniza com backend
```

Esse é o **modo padrão** para usuários.

---

## 3.2 Cloud SaaS

Backend hospedado pelo próprio Metric.

Exemplo:

```
app.metric.dev
```

Esse ambiente oferece:

- autenticação
- painel admin
- controle de organizações
- gestão de planos
- convites

---

## 3.3 Self-Hosted

Empresas podem rodar o Metric em sua própria infraestrutura.

Exemplo:

```
metric.internal.company.com
```

Motivos comuns:

- políticas de segurança
- dados internos
- compliance
- isolamento de rede

---

# 4. Shared Application Layer

Uma decisão importante foi **utilizar a mesma application layer para todos os modos**:

- Cloud
- Desktop
- Self-Hosted

Arquitetura:

```
+----------------------+
|   Desktop Client     |
+----------------------+
           |
           v
+----------------------+
| Application Layer    |
+----------------------+
           |
           v
+----------------------+
|   Data Storage       |
+----------------------+
```

Essa camada contém:

- regras de negócio
- autenticação
- sistema de organizações
- gestão de membros

---

## Motivos da decisão

### 1. Reduzir duplicação de código

Manter múltiplas implementações gera inconsistência.

Uma única application layer evita:

- divergência de regras
- bugs duplicados
- manutenção difícil

---

### 2. Facilitar manutenção

Correções são feitas **uma única vez**.

---

### 3. Garantir paridade entre Cloud e Self-Hosted

Usuários self-hosted terão a **mesma experiência**.

---

# 5. Authentication Model

A autenticação utilizará **Magic Link**.

Usuários não terão senha.

---

## Fluxo

```
Usuário digita email
↓
Servidor envia email com link
↓
Usuário clica no link
↓
Sessão autenticada
```

---

## Motivos da decisão

### 1. Redução de responsabilidade de segurança

Gerenciar senhas implica:

- hashing seguro
- recuperação de senha
- proteção contra ataques
- armazenamento seguro

Magic Link elimina:

- armazenamento de senhas
- vazamentos de credenciais

---

### 2. Simplicidade para usuários

Fluxo extremamente simples:

```
digitar email → clicar no link
```

---

### 3. Menos suporte técnico

Não existem problemas como:

- senha esquecida
- reset de senha
- bloqueio de conta

---

### 4. Adequado para empresas

Em ambientes corporativos o email já é a identidade principal.

```
Organization

   +-- Members
   |
   +-- Admins
   |
   +-- Plan
```

---

## Tipos de usuários

### Owner

- criador da organização
- controla billing
- acesso total

---

### Admin

- gerencia membros
- envia convites
- controla configurações

---

### Member

- utiliza o produto
- acesso limitado

# 7. Admin Panel

Exemplo:

```
metric.dev/panel
```

---

## Funcionalidades

O painel permitirá:

### Organização

- configurações
- plano

### Membros

- adicionar membros
- remover membros
- definir roles

### Billing

- histórico de pagamentos

# 8. Member Onboarding

Usuários entram em organizações via **convite**.

---

## Modelo de convite

Link único:

```
metric.dev/invite/ORG_CODE
```

Exemplo:

```
metric.dev/invite/atk-team-ABC123
```

---

## Fluxo de entrada

```
Admin cria invite link
↓
Admin envia link para equipe
↓
Usuário abre link
↓
Usuário informa email
↓
Recebe Magic Link
↓
Conta vinculada à organização
```

---

## Motivos da decisão

Esse modelo evita:

- cadastro manual de 100 usuários
- digitação manual de emails
- erros administrativos

Além disso, permite onboarding rápido.

---

# 9. Desktop Authentication Flow

Quando um usuário abre o cliente desktop:

```
Aplicação inicia
↓
Usuário clica em login
↓
Aplicação abre browser
↓
Usuário autentica no site
↓
Token retorna para desktop
↓
Sessão ativa
```

Esse fluxo é comum em aplicações como:

- Slack
- Notion
- Figma

---

# 10. Self-Hosted Behavior

No modo self-hosted:

```
https://metric.company.com
```

A organização normalmente é **única**.

Portanto:

- não existe múltiplas organizações
- não é necessário inserir código de org

---

## Configuração

A organização pode ser definida via variável de ambiente:

```
ORG_ID=company
```

---

## Fluxo de login

```
Usuário abre instância interna
↓
Digita email
↓
Recebe Magic Link
↓
Login realizado
```

---

# 11. Licensing Model

Licenças serão gerenciadas no backend do Metric.

Planos disponíveis:

### Free

- open core
- funcionalidades básicas

---

### Pro

Para:

- pequenos times
- profissionais individuais

---

### Enterprise

Para:

- empresas
- controle de equipes grandes
- recursos avançados

---

## Controle de licença

O backend controla:

```
Organization
↓
Plan
↓
Feature Flags
```

Cliente recebe permissões baseadas no plano.

---

# 12. Consequences

## Positivas

- arquitetura simples
- fácil onboarding
- baixo custo operacional
- segurança reduzida em autenticação
- suporte natural a SaaS e self-hosted

---

## Negativas

- dependência de email para login
- onboarding inicial pode exigir download do cliente
- algumas empresas podem preferir SSO completo

---

# 13. Future Improvements

Possíveis evoluções futuras:

- suporte a SSO (SAML / OAuth)
- provisionamento automático de usuários
- SCIM para empresas grandes
- autenticação offline opcional
- gerenciamento avançado de permissões

---

# 14. Summary

A arquitetura escolhida prioriza:

- simplicidade
- escalabilidade
- baixo custo operacional
- facilidade de adoção

O sistema utiliza:

- modelo Open-Core
- autenticação Magic Link
- organizações com membros
- painel administrativo web
- cliente desktop
- suporte opcional a self-hosting

Essa combinação permite atender desde desenvolvedores individuais até empresas com centenas de usuários.
