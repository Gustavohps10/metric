# ADR-002 — Múltiplos Data Sources por Workspace

## Status

Accepted

## Date

2026-03-14

---

# 1. Context

O sistema hoje modela **1 workspace : 1 datasource**. O login e a sincronização estão amarrados a esse modelo, o que dificulta:

- Conectar o mesmo workspace a mais de uma fonte (ex.: Redmine + Jira).
- Ter credenciais independentes por conexão.
- Exibir e operar múltiplas sincronizações de forma clara na UI.

É necessário suportar **N datasources por workspace**, com autenticação e sync por conexão, e deixar preparado o login global do app (sidebar como placeholder).

---

# 2. Decisão

## 2.1 Modelo de dados

- **Workspace** passa a ter **múltiplas conexões** (não mais um único `dataSource`).
- Cada conexão é identificada por:
  - `workspaceId`
  - `dataSourceId` (tipo do plugin, ex.: `redmine`, `jira`, ou ID de instância como `redmine-for-test-2`).
- Credenciais e configuração são **por conexão** (chave `workspace-session-${workspaceId}-${dataSourceId}`).
- Workspace expõe `dataSourceConnections: { id: string; config?: Record<string, unknown> }[]` para compatibilidade com N conexões; mantém-se suporte à leitura do antigo `dataSource` único para migração.

## 2.2 Autenticação e login

- **Login não fica amarrado ao workspace.** Cada **conexão** (datasource) tem sua própria autenticação.
- Se uma conexão estiver inválida (credenciais erradas/expiradas), apenas o **sync daquela conexão** falha; as demais continuam.
- **Sidebar:** o item de “login” fica **fixo/placeholder**, sem ação por enquanto; representa o futuro login global do app (contas Metric, organizações, etc.).
- **Onde o usuário gerencia conexões:** apenas na **página de Plugins** (configurar/addons): cada plugin de datasource exibe suas conexões (e permite conectar/desconectar/editar por conexão).

## 2.3 Sincronização

- Cada (workspace, conexão) possui seu próprio **motor de sync** por coleção (metadata, tasks, timeEntries).
- Checkpoints são **por replicação** (por coleção e por conexão), não compartilhados.
- Endpoints de pull/push já recebem `workspaceId` e `dataSourceId`; mantém-se esse contrato.

## 2.4 UI — Header (Monitor de Sincronização)

- O header mostra o status de sincronização **agrupado por datasource** (não lista plana de todas as replicações).
- Estrutura:
  - **Por datasource:** um bloco (collapsible/toggle) por conexão.
  - Dentro de cada bloco: **Metadata**, **Tarefas**, **Apontamentos** com estado (sincronizado / buscando / erro) e última replicação.
- Exemplo: "Redmine for test 1" → Metadata, Tarefas, Apontamentos; "Jira" → idem.

## 2.5 Fluxo do usuário

1. Usuário entra no app e cria um workspace.
2. Na **página de Plugins**, vê os datasources disponíveis (ex.: Redmine, Jira, Redmine for test 1, Redmine for test 2).
3. Conecta as fontes desejadas; cada uma com seu login/config.
4. Os motores de sync desse workspace são ativados para cada conexão.
5. No **header**, o usuário acompanha o status **agrupado por datasource**, com cada coleção no seu estado.
6. Conexões são gerenciadas apenas na página de Plugins; a sidebar não realiza login de conexão (apenas placeholder para login global futuro).

## 2.6 Cenário de teste — Dois Redmines (clones)

- Dois “Redmine for test” com **IDs distintos** (ex.: `redmine-for-test-1`, `redmine-for-test-2`).
- Na tela de configurar plugins/conectar, o usuário vê as duas opções e pode conectar as duas ao mesmo workspace.
- No header, aparecem dois blocos (um por instância), cada um com Metadata, Tarefas e Apontamentos no seu estado.
- Implementação: duas pastas em `addons/datasource` (ou dois addons) com o mesmo código e IDs diferentes; o resolver usa o `dataSourceId` para credenciais e checkpoints separados.

---

# 3. Consequências

- **Positivas:** workspace pode agregar várias fontes; login por conexão; sync isolado por conexão; UI clara por datasource; testes com múltiplas instâncias do mesmo tipo.
- **Negativas:** modelo de workspace e de “lista de conexões” precisa ser implementado; sidebar deixa de ter ação de login (placeholder até haver login global).

---

# 4. Decisões em aberto e problemas conhecidos

## 4.1 Desvincular (unlink) uma conexão

Ao desvincular uma fonte do workspace, o que fazer com os dados já sincronizados localmente (tasks, time entries, metadata daquele `dataSourceId`)?

- **Opção A — Apagar tudo:** ao desvincular, remover do RxDB todos os documentos em que `dataSourceId` é a conexão desvinculada; limpar checkpoints de replicação dessa conexão. Dados locais deixam de existir; se reconectar depois, sync recomeça do zero.
- **Opção B — Manter local, parar sync:** apenas deixar de sincronizar; dados já baixados permanecem no RxDB (útil para histórico/offline), mas podem ficar desatualizados e ocupar espaço.
- **Opção C — Perguntar ao usuário:** na ação "Desvincular", exibir diálogo: "Manter cópia local dos dados?" (Sim = B, Não = A).

**Recomendação a definir:** A ou C (com default para A) para evitar mistura de dados "órfãos" com outras conexões.

**Problemas relacionados:**

- **Misturar contas:** se o mesmo workspace tiver várias conexões (ex.: dois usuários no mesmo Redmine), ao desvincular uma conta, tasks e apontamentos da outra devem permanecer intactos. O escopo por `dataSourceId` (e chave composta `datasourceId::entityId`) já permite isolar; a decisão acima aplica-se apenas aos documentos da conexão desvinculada.
- **Credenciais:** ao desvincular, as credenciais armazenadas (`workspace-session-${workspaceId}-${dataSourceId}`) devem ser removidas em qualquer opção.

---

## 4.2 Mesmo datasource conectado N vezes (ex.: 2 usuários no mesmo Redmine)

Cenário: um workspace precisa usar o **mesmo** Redmine (mesma instância, mesmo addon) com **duas contas diferentes** (ex.: João e Maria).

- Hoje, cada "conexão" é identificada por `dataSourceId` (ex.: id do addon `redmine-for-test-1`). Conectar o mesmo addon duas vezes com credenciais diferentes exigiria dois IDs de conexão distintos.
- **Decisão em aberto:** introduzir um conceito de **instância de conexão** por workspace:
  - Cada conexão no workspace tem um identificador único (ex.: `connectionId`), não só o tipo/addon (`dataSourceId`). Ex.: `redmine-for-test-1::joao`, `redmine-for-test-1::maria`, ou UUID por conexão.
  - Assim, "conectar o mesmo datasource N vezes" = N conexões com o mesmo `dataSourceType`/addon mas `connectionId` (e credenciais) diferentes.
- **Impacto:** armazenamento de credenciais e checkpoints por `workspaceId + connectionId`; listas de tarefas/apontamentos e filtros na UI devem considerar conexão (e usuário) para não misturar contas.

---

## 4.3 Apontamento (criar time entry): datasource, tarefa e usuário

Com N conexões (e possivelmente N usuários no mesmo datasource), ao **criar um apontamento** o sistema deve deixar explícito:

1. **Para qual conexão (datasource/connection)** o apontamento vai — define em qual fonte o tempo será enviado e com qual contexto de autenticação.
2. **Para qual tarefa** — lista de tarefas deve ser filtrada pela conexão escolhida (e, se aplicável, pelo usuário da conexão).
3. **Em nome de qual usuário (member)** — se a mesma conexão puder representar mais de um usuário (cenário 4.2), ou para consistência com a API remota, o apontamento deve estar associado a um membro (user) daquela conexão.

**Decisões em aberto:**

- Na tela de apontamentos: dropdown ou seletor de "Fonte / Conexão" antes de escolher tarefa; lista de tarefas e de usuários (se necessário) escopada por essa conexão.
- Se houver apenas uma conexão ativa, pode-se defaultar para ela sem exigir seleção explícita; com N conexões, a seleção deve ser obrigatória.

---

## 4.4 Outros problemas identificados

- **Identidade "atual" ambígua:** com N conexões, não há um único "usuário logado" global; cada conexão tem seu próprio membro/credencial. A UI (ex.: header, apontamentos) deve deixar claro em qual contexto (conexão/usuário) a ação está sendo feita.
- **Metadata agregada:** hoje parte da UI usa `db.metadata.findOne()` (um único doc). Com N datasources há N documentos de metadata (`dataSourceId` diferente). Listas (status, prioridades, atividades) precisam ser agregadas ou filtradas por conexão, para não misturar metadados de fontes diferentes.
- **Export/backup:** ao exportar dados do workspace, o formato deve identificar `dataSourceId` (e, no futuro, `connectionId`) por registro, para que importação ou auditoria saibam a origem de cada task/time entry.
- **Conflitos de sync:** em cenários com mesma lógica de negócio em duas conexões (raro), definir política de resolução; hoje o modelo evita conflito ao isolar por `dataSourceId` e checkpoint por replicação.
- **Performance e UX:** muitas conexões no mesmo workspace podem sobrecarregar listas (tarefas/apontamentos). Considerar filtros por conexão e agrupamento por datasource nas telas principais.

---

# 5. Resumo

- **N datasources por workspace:** múltiplas conexões por workspace, cada uma com id (e opcionalmente instância/clone).
- **Login:** desacoplado do workspace; por conexão; gerenciado na página de Plugins; sidebar com placeholder para login global.
- **Sync:** um motor por (workspace, conexão, coleção); checkpoint por replicação; falha de uma conexão não derruba as outras.
- **Header:** status de sync **agrupado por datasource**, com Metadata, Tarefas e Apontamentos por conexão (toggle/collapse por datasource).
- **Teste:** dois Redmines de teste (IDs diferentes) para validar N datasources e exibição agrupada no header.
- **Em aberto:** política ao desvincular (apagar vs manter dados locais); suporte a "mesmo datasource N vezes" (connectionId); apontamento com seleção explícita de conexão, tarefa e usuário; tratamento de metadata e identidade com N conexões.
